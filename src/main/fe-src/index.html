Instructions

The DEVINTA GitHub repository search engine supports querying by a magnitude of parameters. Keep in mind however that all parameters are optional. Searching without any specified parameters yields all the currently existing records in the database.

Repository name matching has two distinct modes:

  • Contains - Which matches all repositories containing the specific keyword. Keywords can only contain alphanumeric characters, along with the dot, slash, dash and underscore characters. Maximum allowed keyword length is 140 characters.
  • Equals - Which matches a repository by fully qualified name: username/reponame. The username is the name of the repository owner, and the reponame is the name of the project. Allowed characters include letters, numbers, dots, dashes and underscores. The maximum allowed length of the username and reponame are 39 and 100 characters respectively.

Repositories can also be searched by their license, main language and a label they use. To aid the user in constructing the query, these input fields use a typeahead to suggest existing values to filter by.

History, activity and popularity filters (number of commits, contributors, stars, watchers...) limit selections according to ranges specified in the query. These selection intervals are fully inclusive, and can be upper bound, lower bound, fully bound and even unbound. Inputs of this type only except positive integers and zero. The up and down arrows next to the input increment and decrement its value.

Date-based filters limit repository selection by date of creation or last commit. The supported date format is YYYY-MM-DD. Clicking on the calendar button next to the input reveals a date picker for simplified selection.

Finally, there are several additional binary filters. These filters allow separation of fork repositories from source projects, selection of projects that have open issues and pull requests, as well as selection of projects that have a wiki and that are licensed. Keep in mind that only one of the following can be selected: "Only Forked Projects" and "Exclude Forked Projects".

How It Works

The mining algorithm is the core of our search engine. In broad terms, the miner is responsible for mining all the public projects written in certain languages, from oldest to newest created. It is important to note that the miner currently prioritises repositories consisting predominantly of code written in more popular languages, which include: Java, Kotlin, C, C++, Python, JavaScript and TypeScript. Since mining is sequentially performed for each language, all the repositories of one language have to be mined before moving to the next one.

The mining algorithm works as follows:

The application retrieves the list of all supported languages from a database table. For each language, the application then checks if any prior mining has been conducted for said language. If no prior mining was conducted for the language, the application mines all repositories created between January 1st 2008 and two hours before the mining process was started. Conversely, the mining is resumed from the date indicated in the database, and the application mines all repositories created and updated between that date and two hours before the mining process was started.

Having selected a time interval for mining a language, there may be more repositories than can be retrieved by the GitHub API. To work around this issue, if a time interval has too many results, then we split it into more manageable chunks.

Interval mining is performed in the following manner:

  1. A “request stack” is defined, used for storing future mining intervals. The first element of the data structure will always be the broad mining interval of a language, described previously in more detail.
  2. We take a date interval from the top of the stack. Said interval is passed to a GitHub API call, requesting all projects and associated forks of a certain language created during that time. If the stack is empty, then the crawl is finished.
  3. Depending on the number of total results returned in the response:
    1. If the number of results is greater than 1000, then the time interval is split in half, and the two new intervals are pushed to the stack. Return to step 2.
    2. Otherwise, we iterate over the result list. For each result:
      1. Scrape the missing information from the repository web-page, and save the full record to the database.
      2. Using the GitHub API, retrieve the repository labels, and save them to the database
      3. Using the GitHub API, retrieve the repository languages, and save them to the database.
    3. After iterating through all the results, return to step 2.

The mined repository information can then be accessed by users of this website through queries. Once a query is submitted, a REST request containing the query parameters is sent to the central server. The server then processes the sent information and propagates a query to the application database. Query results are sent back to the user, consisting of a page of the first twenty alphabetically sorted matches, along with the download links for the full result set, as well as other page navigation links.
